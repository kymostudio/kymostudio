This is a **reference note on prior art**, not a specification of kymo. It captures D2's design choices so the team can consult them when evolving kymo's DSL, layout, and render pipeline. No code or behavior in this repository depends on D2.

## 1. Overview

**D2** is a modern, open-source text-to-diagram scripting language and renderer, built by Terrastruct in **Go** and released under the **Mozilla Public License 2.0**. Upstream describes it as "the fastest and easiest way to get a mental model from your head onto the screen." It occupies the same problem space as Mermaid, Graphviz, and PlantUML â€” declarative source text in, diagram out â€” with a stronger emphasis on (a) layout-engine abstraction, (b) modern stylistic affordances (sketch mode, theming, animation), and (c) a wide catalog of specialized diagram types accessible from the same DSL.

## 2. DSL syntax at a glance

The following snippets are taken from the official tour (<https://d2lang.com/tour/intro>) and are intended to illustrate syntactic style, not to be authoritative grammar.

### 2.1 Connections

```d2
x -> y -> z
```

Directed edges chain with `->`. Reverse with `<-`, bidirectional with `<->`, undirected with `--`.

### 2.2 Containers via dot notation

```d2
server.process.thread
server.cache
server.process -> server.cache
```

Nesting is implicit through the dot path; D2 creates intermediate parent objects on demand. This is markedly terser than kymo's explicit `region { ... }` blocks.

### 2.3 Shapes and inline styling

```d2
db: Database {
  shape: cylinder
  style.fill: "#f0e4ff"
  style.stroke: "#7a3fb8"
  style.sketch: true
}

worker.shape: hexagon
```

Attributes are addressed via the same dot path as objects; `style.*` namespaces purely visual properties. `style.sketch: true` toggles a hand-drawn render that ships in core (no plugin).

### 2.4 Titles, special positions, and nested `style` blocks

```d2
title: Hello Friends {
  near: top-center
  shape: text
  style: {
    font-size: 29
    bold: true
    underline: true
  }
}
x -> y
```

Three things to notice in this snippet (live: [playground link](https://play.d2lang.com/?script=PMsxDsIwEETR3qeYC7ihYwtKxDUCHoSl1Tpab6QElLsjp0j5n2aihlLwoGrD3SutdPwSYJxcEG3OL1rQE9A_00xBcI1RsY3n2ALvZpF7_VJwuR7ybFoE4QuPXKzQtRpP29OeVuQbtvQPAAD__w%3D%3D&)):

1. **`near:` for special positioning.** D2 recognises a fixed vocabulary of anchors (`top-left`, `top-center`, `top-right`, `center-left`, `center-right`, `bottom-left`, `bottom-center`, `bottom-right`). Any object can pin itself to one without participating in the auto-layout â€” useful for titles, legends, and captions. This is more declarative than kymo's current approach of expressing titles through a top-level `title:` directive plus implicit top placement (see `packages/python/src/kymo/to_svg.py`).
2. **`shape: text`.** A shape with no border or fill â€” pure label. kymo today reaches the same effect by using a component without an icon and clearing the stroke; a first-class `text` shape would simplify caption-style nodes.
3. **Nested `style: { ... }` block.** D2 accepts either dot-notation (`style.bold: true`) **or** a nested map (`style: { bold: true }`). The two forms are interchangeable; the block form scales better when many style attributes apply to one element.

### 2.5 Variables, imports, wildcards

D2 supports several composition features that aren't visible at the "Hello world" level:

- **Variables** â€” define once, substitute everywhere:
  ```d2
  vars: {
    primary: "#5563d3"
    radius: 12
  }
  start.style.fill: ${primary}
  ```
- **Imports** â€” split a diagram across files: `...@./shared.d2` spreads the imported object's contents at the current path; `@./shared.d2` imports as a sub-object.
- **Wildcards / globs** â€” bulk-apply styles or properties: `(** -> **)[*].style.stroke: red` styles every edge in the diagram; `*.shape: hexagon` applies to every direct child.
- **Connection references** â€” connections can be addressed and re-styled the same way objects can: `(x -> y).style.stroke: red`.

## 3. Shape catalog

D2 ships with a fixed catalog of named shapes. Beyond these, `shape:` can also take the values of the **specialized diagram types** in Â§4 (`sequence_diagram`, `class`, `sql_table`, `image`, `code`, `text`).

| Category        | Shape keywords |
|-----------------|----------------|
| Default         | `rectangle` (default), `square` |
| Generic         | `page`, `parallelogram`, `document`, `callout`, `cloud` |
| Storage / data  | `cylinder`, `queue`, `stored_data` |
| Workflow        | `step`, `diamond` (decision), `package` |
| Geometric       | `oval`, `circle`, `hexagon` |
| People          | `person`, `c4-person` |
| Specialized     | `text`, `image`, `code`, `class`, `sql_table`, `sequence_diagram` (covered in Â§4) |

Notes:

- Keys are case-insensitive (`shape: Circle` works).
- `circle` and `square` maintain a 1:1 aspect ratio.
- Default is `rectangle` when `shape:` is omitted.

By contrast, kymo's `packages/python/src/kymo/icons.py` provides a different vocabulary (isometric cubes, flat boxes, hex agents, cylinders, circles, etc.) â€” the shape sets are not 1:1 and shouldn't be conflated.

## 4. Specialized diagram types

D2 reuses the same DSL for several diagram families by overloading `shape:`. This is the single biggest scope difference vs kymo today.

### 4.1 Sequence diagrams

Setting `shape: sequence_diagram` on a container turns its children into actors and its connections into messages. **Order matters** in this mode â€” unlike normal D2 objects, elements render in declaration order.

```d2
shape: sequence_diagram
alice -> bob: "What does it mean to be well-adjusted?"
bob -> alice: "The ability to play bridge or golf as if they were games."
```

Sub-features (per the docs):

- **Spans / activation boxes** â€” nested objects under an actor represent active execution periods.
- **Groups / fragments** â€” labelled rectangles around subsets of messages.
- **Notes** â€” nested objects with no outgoing connections act as inline commentary.
- **Self-messages** â€” actor â†’ self loops for internal steps.
- **Lifelines** â€” vertical lines that inherit the actor's `stroke` / `stroke-dash`.

### 4.2 UML classes

`shape: class` turns an object into a UML class box. Field rows carry a visibility prefix (`+` public, `-` private, `#` protected), and method rows put return types after the parameter list.

```d2
D2 Parser: {
  shape: class
  +reader: io.RuneReader
  +readerPos: d2ast.Position
  -lookahead: "[]rune"
  #lookaheadPos: d2ast.Position

  +peek(): (r rune, eof bool)
  +rewind(): void
  +commit(): void
  #peekn(n int): (s string, eof bool)
}
```

### 4.3 SQL tables

`shape: sql_table` produces a table block with typed columns and supports edges between columns to indicate foreign keys (e.g. `orders.user_id -> users.id`). Useful for ERDs from the same source language.

### 4.4 Code & text shapes

- `shape: code` â€” embeds a fenced code block as a node. Syntax highlighting is applied if a language tag is supplied.
- `shape: text` â€” Markdown-rendered text node with no chrome. LaTeX (`$ ... $`) blocks are also accepted in text-bearing shapes.

### 4.5 Image shapes

`shape: image` with an `icon:` URL renders the image as the shape itself (vs `icon:` on a non-image shape, which decorates the shape).

### 4.6 Grid diagrams

`grid-rows` / `grid-columns` on a container switches it from auto-layout to a fixed grid. This is how D2 expresses "I want these in a 2x3 matrix" without resorting to absolute coordinates.

```d2
objects: {
  grid-rows: 2
  grid-columns: 3
  grid-gap: 10
  A; B; C
  D; E; F
}
```

Additional controls: `width`, `height`, `vertical-gap`, `horizontal-gap`. With both rows and columns set, the first-declared dimension dominates the fill order. `grid-gap: 0` enables seamless table-like layouts.

This is conceptually similar to kymo's per-row height sync (`packages/python/src/kymo/layout.py`) but exposed as a first-class DSL concept rather than emerging from the layout algorithm.

## 5. Composition: layers, scenarios, steps

D2 treats a single source file as a *collection of boards* rather than a single diagram. Three composition keywords differ in how they inherit from each other:

| Keyword     | Inherits from           | Use case                                             |
|-------------|-------------------------|------------------------------------------------------|
| `layers`    | Nothing (new base)      | Independent views sharing a file                     |
| `scenarios` | The base layer          | "What-if" branches (e.g. normal vs hotfix deploy)    |
| `steps`     | The previous step       | Sequential narratives, progressive disclosure        |

```d2
direction: right
github -> aws
layers: {
  detailed: {
    github.actions -> aws.ec2
  }
}
scenarios: {
  hotfix: {
    github.style.stroke: red
    github -> aws: "skip review"
  }
}
```

Boards become navigable tabs in the SVG output and slide-style frames in the animated SVG. kymo has no parallel of this today â€” `samples/` files are single-board.

## 6. Themes

D2 has a substantial set of built-in themes including **Grape soda**, **Vanilla nitro cola**, **Mixed berry blue**, and **Terminal** (uppercase labels, no border radius, monospaced font, dotted fill). Selection is via CLI flag:

```bash
d2 --theme=200 in.d2 out.svg                 # light theme
d2 --theme=200 --dark-theme=300 in.d2 out.svg # adaptive dark mode
```

Dark themes are off by default; users opt in via `--dark-theme` for `prefers-color-scheme` adaptation. kymo currently has no theme system â€” accent colours are per-component and the palette is hand-coded.

## 7. Icons

```d2
gcloud: { icon: https://icons.terrastruct.com/gcp/Product%20Icons/Compute/Compute%20Engine.svg }
my_cat: { icon: ./my_cat.png }
```

D2 has **no bundled icon library** â€” `icon:` accepts any URL or local path. The maintainers separately host a free catalog at <https://icons.terrastruct.com> tuned for software-architecture diagrams (AWS, GCP, Azure, k8s, etc.). Icon placement is automatic and engine-aware so labels aren't overlapped.

kymo bundles its own icon set in `packages/python/src/kymo/icons.py` and renders them inline as SVG â€” no network fetch, no external file references. Different tradeoff: tighter set, zero network dependency, opinionated style.

## 8. Interactive features

```d2
gcloud: {
  tooltip: Google Cloud Platform.
  link: https://cloud.google.com
}
```

D2 supports two interactive attributes on any object:

- **`tooltip:`** â€” text shown on hover. In static raster exports (PNG/PDF), tooltips degrade to numbered icons with an appendix.
- **`link:`** â€” clickable hyperlink. URLs containing `#` must be quoted to avoid being parsed as comments.

These work in the SVG output natively (browser-rendered) and degrade gracefully in raster targets.

## 9. Layout engines

D2 deliberately abstracts layout away from the language. Three engines are supported:

| Engine | Bundled | Strength                                     |
|--------|---------|----------------------------------------------|
| Dagre  | âœ“ (default) | Hierarchical / layered graphs, fast      |
| ELK    | âœ“       | Node-link with directional flow, denser graphs |
| TALA   | Separate install (Terrastruct) | Tuned for software architecture diagrams |

Selection is a CLI flag (`d2 --layout=elk in.d2 out.svg`). The architectural takeaway: **the DSL is decoupled from the layout strategy**. kymo's `packages/python/src/kymo/layout.py` currently hardcodes one strategy (per-row height sync + tight cell sizing); a pluggable interface would let us experiment with ELK-style flows without forking the parser.

## 10. Output formats

D2 emits **SVG**, **PNG**, **PDF**, and **animated SVG**. Upstream markets itself as *"the only language that can produce animated diagrams from text"* â€” though that claim should be read in the context of late-2024 / 2025; kymo's pipeline (animated SVG plus animated WebP via frame synthesis in `packages/python/src/kymo/to_webp.py`) is a counter-example.

Animation models differ:

- **D2**: declares animations at the SVG-element level (e.g. `style.animated: true` on a connection produces a moving dash). Output is a single SVG with embedded `<animate>` / SMIL or CSS-driven motion. Plays natively in browsers; does not require a rasterized fallback.
- **kymo**: declares animations at the source DSL level and emits both an animated SVG **and** a synthesized animated WebP (30 frames over 1.2 s, rasterized via `resvg`, stitched with Pillow). The WebP path exists for environments where SVG animation is unreliable (chat clients, Markdown previewers, slide decks).

If kymo ever wanted to consume D2-style animation declarations, the WebP frame synthesizer would need to inspect SVG `<animate>` elements rather than the project's bespoke `flow` markers.

## 11. Library / SDK mode

D2 ships as both a **CLI** and a **Go library**, enabling programmatic diagram generation without shelling out. Community SDKs exist for **Python**, **C#**, and **JavaScript** (links via the upstream README). For kymo, the practical implication is:

> If we wanted to keep kymo's user-facing DSL but offload layout and SVG rendering to D2, a Python wrapper around the D2 binary or a CFFI binding into the Go library would be the integration surface. This is **not** currently on the roadmap; it is noted here purely for future reference.

## 12. Ecosystem and adopters

Upstream README lists production users including **Elasticsearch**, **Sourcegraph**, **Temporal**, and **Tauri**. The ecosystem includes a VS Code extension, a hosted online editor at <https://play.d2lang.com/>, and an Obsidian plugin. Mention here is to signal maturity; none of this is a dependency for kymo.
