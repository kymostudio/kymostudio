This document isolates the **prior-art comparison** between [D2](https://d2lang.com/) and kymo. The factual reference (DSL syntax, shape catalog, layout engines, output formats) lives in [`d2.md`](d2.md); read that first if you need ground truth on how D2 actually behaves.

The comparison is kept separate so it can evolve at a different cadence than the factual reference: it is **an opinion shaped by kymo's current direction**, not a description of an external tool. Update it when kymo's DSL, layout, or render pipeline changes — even if upstream D2 does not. Of all the tools surveyed in `docs/softwares/`, **D2 is the closest peer to kymo**: both are text-to-diagram languages with an owned render pipeline, so this is a same-category, like-for-like comparison (unlike the BPMN engines or WYSIWYG editors).

## 1. At-a-glance matrix

| Axis | D2 | kymo |
|------|-----|------|
| Primary purpose | General text-to-diagram language | Render static architecture diagrams |
| Implementation language | Go | Python 3 (+ JS/TS data-model port) |
| License | MPL-2.0 | Apache-2.0 |
| DSL style | Minimal — `->` edges, dot-nested containers | Declarative blocks — `region { ... }`, `component`, `→` |
| Container model | Implicit, via dot paths | Explicit, via named `region` blocks with padding / outer label |
| Layout | Pluggable: Dagre (default) / ELK / TALA | Single algorithm: grid pack + per-row height sync |
| Grid layouts | First-class (`grid-rows` / `grid-columns`) | Emergent from layout algorithm |
| Edge routing | Engine-dependent | Bespoke orthogonal router with rounded corners |
| Diagram types | Block, sequence, UML class, SQL, code, image, grid | Block / architecture diagrams only |
| Composition | Multi-board: `layers` / `scenarios` / `steps` | Single-board per file |
| Animation | Native animated SVG (`style.animated`) | Animated SVG + frame-synthesized WebP |
| Output | SVG, PNG, PDF, animated SVG | SVG, animated SVG, animated WebP, Figma, Excalidraw |
| Themes | Many built-in + dark mode | None — accents hand-coded |
| Icons | Per-element URL/file; external catalog (icons.terrastruct.com) | Bundled SVG library |
| Interactivity | `tooltip:` and `link:` per object | None (static output) |
| Embedding | Go lib; community Python/JS/C# SDKs | Python module (CLI entry) |
| Styling form | Dot-notation `style.x` or nested `style: { }` | YAML-like keys per component |

## 2. Headline tradeoffs

### 2.1 Two external grammars, two different bets

D2 and kymo made the same foundational choice — an external DSL with its own parser, not a host-language API (the bet `mingrammer/diagrams` made; see [`diagrams.mingrammer.comparision.md` §2.1](diagrams.mingrammer.comparision.md)). That shared choice is why this comparison is so close: both pay for their own parser once and in return control their syntax and own their tooling destiny. The divergence is in *scope of ambition*: D2 is a general-purpose diagramming language (block, sequence, UML, SQL, grid, …) while kymo is deliberately narrow (architecture diagrams with an opinionated icon set). D2 optimises for breadth; kymo optimises for a tight, finished look in one domain.

### 2.2 Layout: pluggable vs owned

D2 abstracts layout behind a CLI flag — Dagre, ELK, or TALA — so the language is decoupled from the layout strategy. kymo hardcodes one algorithm (grid pack + per-row height sync in `packages/python/src/kymo/layout.py`) but in exchange owns the H-V-H midpoint Z edge default ([[feedback-kymo-edge-routing]]) that the team specifically wants and that no generic engine produces. The tradeoff is real both ways: D2 can trial radically different layouts for free; kymo gets exactly the routing aesthetic it designed for and nothing it didn't.

### 2.3 Breadth of diagram types as D2's moat

D2's biggest lead is the catalog of diagram families reachable from one language: `shape: sequence_diagram`, `class`, `sql_table`, `code`, `grid`. kymo draws block/architecture diagrams only. This is the same shape of gap kymo has against `diagrams`' node catalog — a breadth advantage that rewards accumulated work, not a capability kymo can close cheaply. The honest stance: don't chase D2's type breadth; stay excellent at the one diagram class kymo targets.

### 2.4 Composition and themes — borrowable, not yet present

D2's `layers`/`scenarios`/`steps` multi-board model and its theme system (incl. dark mode) are features kymo has no parallel for today. Both are genuinely borrowable (see §4) and neither requires abandoning kymo's design centre. kymo's counter-strength is animation: it ships both animated SVG **and** a frame-synthesized WebP (`packages/python/src/kymo/to_webp.py`) for environments where SVG animation is unreliable — a path D2 does not take.

### 2.5 Ecosystem age

D2 (≈23.7k stars, VS Code extension, hosted playground, Obsidian plugin, SDKs in three languages) carries years of adoption; kymo is early. As with `diagrams`, this is a fact of project age, not a design defect — but it compounds with the theme/composition gaps to make D2 the more "complete" tool today.

## 3. Detailed scoring by category

The matrix in §1 says *what* differs; this section grades *how well* each tool handles each dimension, using the same five categories as [`diagrams.mingrammer.comparision.md`](diagrams.mingrammer.comparision.md) (D2 is the same tool class, so the rubric transfers directly). The per-category totals roll up to an overall in §3.6.

**Scale (per cell, out of 10):**

| Range | Meaning |
|:-:|---|
| 9–10 | Industry-leading; little room to improve. |
| 7–8  | Good; minor gaps that don't bite in practice. |
| 5–6  | Adequate; works but has known limits. |
| 3–4  | Limited; users routinely hit the ceiling. |
| 1–2  | Absent or unusable. |

**Caveats.** Scores for `kymo` reflect what is observable in this repo as of 2026-05-21 (`packages/python/src/kymo/`, `icons/`, `samples/`, `showcase/`, the layout-tree + Figma/Excalidraw exporters) and are held **identical to the kymo column in [`diagrams.mingrammer.comparision.md`](diagrams.mingrammer.comparision.md)** so the two same-category comparisons stay consistent. Scores for `D2` reflect upstream 0.7.1 as documented at <https://d2lang.com/> on 2026-05-18. The **Why** column is load-bearing; numbers without it are worthless — do not strip.

### 3.1 Category A — DSL & Syntax

| # | Criterion | D2 | kymo | Why |
|---|-----------|:--:|:----:|-----|
| A1 | Concision on small diagrams | 9 | 7 | D2's `x -> y -> z` and dot-nested containers are about as terse as a diagram language gets; kymo's `region { ... }` blocks are clean but more verbose. |
| A2 | Safety (no precedence/parse footguns) | 8 | 9 | Both own their grammar (no host-language pitfalls); kymo's block grammar is marginally less ambiguous than D2's overloaded `shape:`/dot-path scheme. |
| A3 | Extensibility (vars, imports, reuse) | 6 | 3 | D2 has `vars`, `...@import`, wildcards/globs; kymo has no variable/import/repeat construct today. |
| A4 | Grouping/container semantics | 6 | 7 | D2 containers are implicit via dot paths (no kind); kymo's typed `region` containers drive layout/styling decisions. |
| | **Category total / 40** | **29** | **26** | **D2 +3** — D2's reuse features (A3) outweigh kymo's container-semantics edge. |

### 3.2 Category B — Layout & Edges

| # | Criterion | D2 | kymo | Why |
|---|-----------|:--:|:----:|-----|
| B1 | Default layout quality | 8 | 8 | Dagre default is strong; kymo's first-party engine is tuned for architecture diagrams. A wash. |
| B2 | User layout control | 8 | 8 | D2 offers grid/`near`/engine choice; kymo exposes a first-class layout-tree DSL. Different routes, similar power. |
| B3 | Edge routing aesthetic | 6 | 10 | D2's routing is engine-dependent and not the H-V-H midpoint Z the team wants ([[feedback-kymo-edge-routing]]); kymo defaults to exactly that. |
| B4 | Edge styling | 8 | 5 | D2 has rich per-edge `style.*` incl. `animated`; kymo has labels but a smaller per-edge styling surface. |
| | **Category total / 40** | **30** | **31** | **kymo +1** — kymo's owned Z-routing offsets D2's richer edge styling. |

### 3.3 Category C — Icons & Catalog

| # | Criterion | D2 | kymo | Why |
|---|-----------|:--:|:----:|-----|
| C1 | Catalog breadth | 6 | 3 | D2 ships no bundled catalog but maintains a free external one (icons.terrastruct.com) tuned for architecture; kymo has a single project-managed `icons/`. |
| C2 | Icon escape hatch (arbitrary asset) | 9 | 3 | D2's `icon:` takes any URL/local path; kymo has no documented arbitrary-asset hatch yet. |
| | **Category total / 20** | **15** | **6** | **D2 +9** — D2's "any URL" icon model plus the hosted catalog is a clear lead. |

### 3.4 Category D — Output & Interop

| # | Criterion | D2 | kymo | Why |
|---|-----------|:--:|:----:|-----|
| D1 | Output format breadth | 8 | 6 | D2 emits SVG/PNG/PDF/animated-SVG; kymo is SVG-first plus animated WebP and Figma/Excalidraw exporters (no PNG/PDF yet). |
| D2 | Round-trip with editors | 3 | 6 | D2 is source-to-image only; kymo's Figma/Excalidraw exporters are one-way today but provide a real bridge. |
| D3 | Cross-language portability | 8 | 8 | D2 is a Go lib with Python/JS/C# SDKs; kymo ships a Python source-of-truth **and** a JS/TS data-model port. A wash. |
| | **Category total / 30** | **19** | **20** | **kymo +1** — kymo's exporter bridge and JS port edge out D2's wider raster formats. |

### 3.5 Category E — Tooling & Ecosystem

| # | Criterion | D2 | kymo | Why |
|---|-----------|:--:|:----:|-----|
| E1 | LSP / IDE / formatter / pkg | 8 | 3 | D2 has a VS Code extension, hosted playground, and formatter; kymo's tooling is early (CLI only). |
| E2 | Maintenance / community gravity | 9 | 3 | D2 has ≈23.7k stars and named production adopters (Elasticsearch, Sourcegraph, Temporal, Tauri); kymo is an early in-house tool. |

### 3.6 Summary

**Weighting rule.** Every one of the **15 criteria** carries **equal weight** (`1/15` of the overall). Category sub-totals are shown for *shape*, not for weighting.

#### 3.6.1 Overall (equal weight per criterion)

| Tool | Sum of 15 cells / 150 | Mean per criterion / 10 | Percentage |
|---|:-:|:-:|:-:|
| `D2`   | **110** | **7.33** | **73 %** |
| `kymo` | **89**  | **5.93** | **59 %** |
| Gap (D2 − kymo) | 21 | 1.40 | 14 pp |

#### 3.6.2 Per-category sub-totals (context only)

| Category | # criteria | Max | D2 | kymo | Δ (kymo − D2) |
|---|:-:|:-:|:-:|:-:|:-:|
| A — DSL & Syntax        | 4 | 40 | 29 | 26 | **−3**  |
| B — Layout & Edges      | 4 | 40 | 30 | 31 | **+1**  |
| C — Icons & Catalog     | 2 | 20 | 15 | 6  | **−9**  |
| D — Output & Interop    | 3 | 30 | 19 | 20 | **+1**  |
| E — Tooling & Ecosystem | 2 | 20 | 17 | 6  | **−11** |
| **Overall**             | **15** | **150** | **110** | **89** | **−21** |

#### 3.6.3 Sensitivity: equal weight per category

If each *category* (not each criterion) were given equal weight (1/5 each), the overall becomes the mean of the five normalised category scores:

| Tool | A | B | C | D | E | Mean / 10 | Percentage |
|---|:-:|:-:|:-:|:-:|:-:|:-:|:-:|
| `D2`   | 7.25 | 7.50 | 7.50 | 6.33 | 8.50 | **7.42** | **74 %** |
| `kymo` | 6.50 | 7.75 | 3.00 | 6.67 | 3.00 | **5.38** | **54 %** |
| Gap | | | | | | 2.04 | **20 pp** |

Equal-per-category *widens* the gap (14 pp → 20 pp) because the two small categories where D2 leads (C and E) get the same say as the larger ones. Use this only if you believe "catalog" and "tooling" each matter as much as "DSL" wholesale — for most kymo readers the per-criterion number (§3.6.1) is the honest one.

#### 3.6.4 Read it this way

- **Headline (§3.6.1): D2 7.33/10 vs kymo 5.93/10 — a 1.4-point / 14-pp gap.** D2 is the more complete tool; kymo is competitive specifically on layout/edges and output bridges.
- **Where kymo holds its own:** B (edge routing, +1) and D (exporter bridge + JS port, +1). Both come from things kymo *owns* — the same pattern as against `diagrams`.
- **Where kymo trails:** C (−9) and E (−11) — icon breadth/escape-hatch and ecosystem maturity. These reward accumulation (catalog content, community-years), not a single fix.
- **Cheapest move on the board:** criterion C2 — an icon escape hatch (load arbitrary SVG/PNG by path), worth ≈ +6 in that cell. It is the same top recommendation as the `diagrams` comparison, which is a strong signal it is the real gap.
- **Don't chase A3/diagram-type breadth.** D2's reuse and multi-type features are nice-to-haves; kymo's value is a tight, finished architecture-diagram look, and B3 (Z-routing) shows what owning the pipeline buys.

### 3.7 Re-score triggers

Re-run the relevant categories when any of the following happens — flag the date and which criteria moved:

1. kymo lands an icon escape hatch (C2) or expands the in-tree catalog meaningfully (C1).
2. kymo adds variables/imports/repeat or a host-language interop (A3).
3. kymo ships an LSP, formatter, or VS Code extension (E1).
4. kymo's exporters become bidirectional (D2), or kymo gains PNG/PDF output (D1).
5. kymo gains a theme system or multi-board composition (would add new criteria / shift A4, B4).
6. Upstream D2 changes default layout engine, adds a new diagram type, or relicenses (B1–B3, C, D).

## 4. Open questions for kymo

These follow from the comparison and the borrowable ideas catalogued in [`d2.md` §14](d2.md):

1. **Pluggable layout interface?** Abstracting `packages/python/src/kymo/layout.py` behind a strategy interface would let kymo trial ELK-style layered routing on dense diagrams without rewriting the parser — at the risk of diluting the owned Z-routing default (B3). Worth a spike before committing.
2. **A theme primitive?** Even two or three named themes (light / dark / print) would close the most visible gap vs D2's theme system, and maps cleanly onto the Figma "variable modes" idea (see [`figma.comparision.md`](figma.comparision.md)).
3. **Multi-board composition?** D2's `layers`/`scenarios`/`steps` would let one `.kymo` file produce several related views (happy-path vs failure) instead of duplicating source. Does kymo's single-board model need this, or is it scope creep?
4. **`tooltip:`/`link:` interactivity and a `text` shape?** Cheap additions to SVG output that D2 already has; useful when the diagram is read in a browser.
5. **Variables/imports?** Would cut duplication across `samples/` when shared component definitions reappear (A3).
