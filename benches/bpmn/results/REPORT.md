## 3. How it is measured (offline scripts)

Three layers, all run by hand and snapshotted — none gate CI on its own:

- **Corpus regression** — `packages/python/tests/_bpmn_regress.py` renders every
  MIWG `.bpmn` and locks `(status, n_nodes, n_edges, SVG-hash)` against a
  committed baseline (`baseline.json` 120-file subset; `baseline_full.json` 840-file
  full corpus). A byte-level change is a regression.
- **Conformance** — `packages/python/tests/test_bpmn_conformance.py` locks that the
  Python and JS importers/exporters agree on the same canonical model
  (`conformance/golden/`); `known_divergences.json` lists any that don't.
- **Timing** — `benches/bpmn/perf.py` (this run) times parse + render over the
  vendored corpus.

`quality.py` only *reads* the first two snapshots — it never re-renders — so the
correctness table reflects the snapshots exactly as committed.

## 4. Element coverage

The importer/exporter maps 30 BPMN 2.0 element types plus
9 event-definition glyphs, against the normative
`docs/formats/bpmn/kymo-mapping.md` (BPMN-MAP-001), which mirrors the OMG BPMN
2.0.2 spec (ISO/IEC 19510:2013): events, tasks (incl. typed tasks), gateways,
sub-processes, pools/lanes/groups, data objects/stores, text annotations, and
sequence/message/association flows.

## 5. Honest limitations

- The 11 *empty-output* files carry no Diagram-Interchange
  geometry; the importer skips them by design (not an error — error count is
  **0**).
- Performance numbers depend on the host above; treat them as a same-machine
  reference, not an absolute SLA.
- Correctness numbers are a snapshot of the committed baselines at run time;
  regenerate the baselines (with the suites' `KYMO_UPDATE_*` flags) before
  re-running this bench to reflect renderer changes.
